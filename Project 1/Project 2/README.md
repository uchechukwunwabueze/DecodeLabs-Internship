# Backend Development Project 2: Database Integration CRUD

This is a small working API for the assignment in `Backend Development P2.pdf`.

It stores users permanently in a SQLite database and supports the full CRUD lifecycle:

- Create a user with `POST /users`
- Read all users with `GET /users`
- Read one user with `GET /users/{id}`
- Update a user with `PUT /users/{id}` or `PATCH /users/{id}`
- Delete a user with `DELETE /users/{id}`
- Reject duplicate emails with HTTP `409 Conflict`

## User Schema

The database table is called `users`.

| Field | Type | Purpose |
| --- | --- | --- |
| `id` | integer | Unique primary key generated by the database |
| `email` | text | User email; must be unique |
| `age` | integer | User age; must be `0` or higher |
| `is_active` | boolean | Whether the account is active |
| `created_at` | text timestamp | When the record was created |

## How to Run

Open a terminal in this folder and run:

```bash
python server.py
```

The API starts at:

```text
http://127.0.0.1:8000
```

The SQLite database file will be created automatically as `users.db`.

## Quick Manual Test

Create a user:

```bash
curl -i -X POST http://127.0.0.1:8000/users \
  -H "Content-Type: application/json" \
  -d "{\"email\":\"ada@example.com\",\"age\":24}"
```

Try the same request again. It should return:

```text
409 Conflict
```

Read users:

```bash
curl -i http://127.0.0.1:8000/users
```

Update a user:

```bash
curl -i -X PATCH http://127.0.0.1:8000/users/1 \
  -H "Content-Type: application/json" \
  -d "{\"age\":25,\"is_active\":false}"
```

Delete a user:

```bash
curl -i -X DELETE http://127.0.0.1:8000/users/1
```

## Run Automated Tests

```bash
python -m unittest test_api.py
```

The test checks the same flow shown in the PDF:

1. `POST` creates a user and returns `201 Created`.
2. Repeating the same `POST` returns `409 Conflict`.
3. `GET` confirms the user was saved.
4. `PATCH` updates the user.
5. `DELETE` removes the user and returns `204 No Content`.

## What This Project Is About

The PDF is teaching database persistence.

In the previous stage, an API may have stored data only in memory. That means if the server restarts, all data disappears. In this project, the API writes data to a database. That means the data survives after the program stops.

The basic flow is:

```text
Client request -> Backend API -> Database -> Backend response
```

For example:

1. A client sends JSON: `{"email":"ada@example.com","age":24}`.
2. The backend validates the JSON.
3. The backend runs a database insert.
4. The database saves the record to disk.
5. The backend returns the saved user with status `201`.

## Beginner Topic Map

Study these topics to understand this assignment deeply:

- Backend development: how servers receive requests and send responses.
- HTTP: methods like `GET`, `POST`, `PATCH`, `DELETE` and status codes like `200`, `201`, `204`, `400`, `404`, `409`.
- REST APIs: naming routes around resources such as `/users`.
- JSON: the data format clients and servers commonly exchange.
- Databases: permanent storage for application data.
- SQL: the language used to create tables, insert rows, update rows, and query rows.
- Schema design: deciding fields, types, constraints, and defaults.
- CRUD: create, read, update, delete.
- Data validation: checking input before saving it.
- Data integrity: keeping the database correct with constraints like `UNIQUE`, `NOT NULL`, and `CHECK`.
- Duplicate prevention: using a unique email field and returning `409 Conflict`.
- Persistence: saving data to disk instead of temporary memory.
- Testing APIs: using Postman, curl, or automated tests to prove the API works.

## How This Version Maps to the PDF
